==================================================
1. GLOBAL PRODUCT RULE: NEXT BEST STEP
==================================================

This is now a core UX rule for the entire app:

The app must always guide the user to the next logical valid step automatically whenever possible.

Principles:
- reduce taps
- reduce cognitive load
- never leave the user stuck after completing an obvious step
- pre-open the next relevant UI state
- make the flow feel assisted, intuitive, and continuous

Example:
When the user selects a departure date, the UI must immediately:
- switch focus to return date
- advance the calendar to the next relevant month if needed
- make it visually clear that the user is now choosing the return date

This principle should be applied across the app wherever the next best step is obvious.

3. ITINERARY INTELLIGENCE FIX
==================================================

Current issue:
The itinerary behaves like a flat list of nice places instead of a realistic travel day.
Example: the same night can show 2 dinner restaurants.

This must be fixed WITHOUT redesigning the UI.

Implement a lightweight post-processing composition layer AFTER candidate retrieval.

Every day must be organized into these logical slots:
- morning
- lunch
- afternoon
- dinner
- late_night (optional)

Slot rules:
- morning: 1 activity only
- lunch: 1 restaurant only
- afternoon: 1 activity only
- dinner: 1 restaurant only
- late_night: 1 bar / nightlife item optional

Never allow:
- 2 restaurants in dinner/night
- 2 restaurants in lunch
- 2 main activities in the same slot

Expected behavior:
Correct:
- Dinner: Oro
- Late night: Bar dos Descasados

Wrong:
- Oro
- Gero

Important:
- Do not change Supabase schema for this fix
- Do not redesign the interface
- Just fix itinerary intelligence with the simplest safe composition layer
- The system must reorganize the places already returned by type/function before showing the itinerary

5. CARD IMAGE ACCURACY FIX
==================================================

This is critical.

Images must correspond to the actual place, or at least to the correct immediate context.

Current observed behavior:
- Forte de Copacabana -> correct
- Barzinho -> correct
- Academia dos Flintstones / Arpoador -> wrong, showing generic Ipanema image

This is not acceptable.

Required image logic priority:
1. exact place image
2. if exact place image is unavailable, use real immediate geographic context
3. never fall back to a generic “nice image” from a nearby but incorrect area if it weakens trust

For example:
For Academia dos Flintstones / Arpoador:
acceptable fallback:
- that exact academy
- Arpoador
- Arpoador rocks / beach / immediate environment

not acceptable fallback:
- generic Ipanema image that feels unrelated

Please fix image resolution/fallback logic so cards feel trustworthy and geographically coherent.
