GOAL

When the user clicks the existing “Create itinerary with AI” CTA inside the Trip screen,
the app must reuse the existing itinerary flow, but prefilled with the user’s saved places and destination context.

This is NOT a new feature from scratch.
This is an integration between:
1. Trip screen saved places
2. existing itinerary creation flow
3. Supabase structured metadata

---

EXPECTED USER EXPERIENCE

Current desired behavior:

- User is on “Trip / Minha Viagem”
- User already has saved places
- User taps “Create itinerary with AI”
- App opens the existing itinerary flow
- Destination step is skipped or prefilled
- Saved places are injected into the flow
- Existing preference steps remain available
- Generated itinerary prioritizes saved places first
- Existing itinerary result UI is reused

Do not change the look and feel of any of these screens.

---

DATA SOURCE

- id
- name
- city / destination
- category

- neighborhood / area if available
- latitude
- longitude
- image

- any relevant metadata already stored in Supabase

If some fields are missing, degrade gracefully without breaking the flow.

ENTRY POINT BEHAVIOR

When user taps the Trip CTA:
- detect current saved places
- infer destination from saved places if possible
- open the existing itinerary flow in “prefilled mode”

Prefilled mode rules:
- skip “Where are you going?” if destination is already known
- skip empty onboarding if user already has saved places
- preserve existing preference selection screens
- update copy only if necessary to reflect saved places context
- do not create duplicate routes or duplicate screens

---

Create a dedicated internal mode for the existing itinerary flow, for example:
- source = "trip_saved_places"

In this mode:
- destination is prefilled from saved places
- savedPlaces[] is injected into flow state/context
- user still chooses preferences such as inspiration / vibe / budget if those screens already exist
- the final itinerary generator receives both:
  1. saved places
  2. user preferences

---

GENERATION LOGIC

Implement itinerary generation with this priority:

1. PRIORITIZE SAVED PLACES
Saved places are the main source of truth.
Do not generate a generic itinerary ignoring them.

2. ORGANIZE, DO NOT RANDOMLY INVENT
The system should organize intelligently based on:
- proximity
- category
- tags
- time-of-day suitability
- daily balance

3. GROUP BY AREA
Cluster saved places by neighborhood / geographic proximity when possible.

4. MATCH TIME OF DAY
Prefer:
- morning: experiences, museums, outdoor, low-friction starts
- lunch: restaurants
- afternoon: lighter experiences, walking, viewpoints, beach, culture
- night: restaurants, bars, nightlife, dinner-oriented suggestions

5. USE TAGS
Use Supabase tags to align with user-selected preferences.

Examples:
- gastronomy → prioritize restaurant and food-related tags
- culture → museums, landmarks, historical places
- relax → beach, nature, calm experiences
- adventure → active experiences
- premium / sophisticated → premium tags, upscale places
- family → family-friendly tags
- friends → social / vibrant / group-suitable places

6. BALANCE EACH DAY
Avoid overload.
Target a reasonable number of items per day.
Prefer 3 to 5 useful stops max depending on density and trip style.

7. OPTIONAL GAP FILLING
Only if current logic already supports it or if easy to add safely:
- when there are too few saved places, allow optional complementary suggestions from Supabase
- complementary suggestions must match destination + proximity + tags
- saved places always come first

Do not let complementary suggestions overpower saved places.

---

MINIMUM FALLBACK RULES

If savedPlaces.length === 0:
- keep current behavior or empty state
- do not break existing flow

If savedPlaces.length < 2:
- allow generation
- but show a lightweight message such as:
  “Add more saved places to build a richer itinerary.”

If destination cannot be inferred:
- fall back to the existing destination step

If tags are missing:
- use category + area + proximity only
