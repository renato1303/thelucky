We need a HARD VALIDATION LAYER for the itinerary output.

This is now the highest priority.

CRITICAL PRODUCT RULE

The Lucky Trip must NEVER suggest places that are not in Supabase.
Supabase is the single source of truth.

If the app is currently suggesting anything not present in Supabase, that is a critical product violation.

We also need smarter temporal validation:
not every restaurante belongs only to almoço/jantar.
Some places are valid for breakfast or brunch, such as:
- Empório Jardim
- João Padeiro

So the validation layer must be strict, but not naive.

CRITICAL RULES

1. Do NOT implement UI changes
2. Do NOT redesign the screen
3. Do NOT change map logic
4. Do NOT add new product features
5. Focus only on the final itinerary validation layer
6. Do NOT let any item survive to the final UI if it violates source-of-truth or temporal plausibility
7. Do NOT invent any place outside Supabase

OBJECTIVE

Design a final validation layer that runs after itinerary assembly and before UI rendering.

This layer must validate:

1. SOURCE-OF-TRUTH INTEGRITY
- every item in the final itinerary must exist in Supabase
- every item must have valid id + source_table
- if an item cannot be resolved to a real Supabase record, it must be flagged
- no invented / hallucinated / orphan items may remain

2. CATEGORY / TIME PLAUSIBILITY
Validate item timing using smarter rules:
- sunset items cannot appear too early
- theater/show items cannot appear in morning
- nightlife/bar/samba cannot appear in morning
- breakfast/brunch places may appear in morning
- lunch-focused restaurants should appear around almoço
- dinner-focused restaurants should appear in noite
- not all restaurantes are equal

3. SPECIAL RESTAURANT EXCEPTIONS
The layer must support cases like:
- Empório Jardim → breakfast / brunch valid
- João Padeiro → breakfast valid

So validation must use:
- momento_ideal
- tags
- especialidade
- tipo
- titulo/name
- any existing structured metadata already available
